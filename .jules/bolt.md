## 2024-07-04 - Safe Set tracking with Object IDs in Crystal Compiler
**Learning:** When using `Set` to track uniqueness of `Type` objects in the Crystal compiler for performance optimizations, relying on `object_id` (or `opaque_id` which might be incorrectly hallucinated) can be problematic. The safe approach is to either store the `Type` itself in the Set (e.g. `Set(Type)`) since they use reference-based hashing and equality, or if you must use integer IDs to ensure exact reference matches and avoid `==` overrides, carefully use `type.object_id` alongside `Set(UInt64)`. Ensure you don't inadvertently create syntax errors by calling non-existent properties on types during tracking.
**Action:** Always verify object property names (like `object_id` vs `opaque_id`) before implementing tracking Sets, and default to storing objects directly in Sets if strict reference equality isn't absolutely demanded by the architecture or if custom `#==` implementation might break logic.
